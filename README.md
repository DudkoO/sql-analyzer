# Sql Analyzer

Этот разработан в целях обучения и представляет собой наглядную реализацию
паттерна MapReduce для анализа SQL запросов.

## Технический стек

- **Node.js**: Высокая производительность и асинхронная обработка запросов.
- **TypeScript**: Статическая типизация для повышения надежности кода.
- **RabbitMQ**: Надежная система обмена сообщениями для распределенной обработки с поддержкой очередей кворума для
  повышения отказоустойчивости.
- **PostgreSQL**: Мощная реляционная база данных с поддержкой сложных запросов и транзакций.
- **Docker**: Контейнеризация для упрощения развертывания и управления зависимостями.
- **Jest**: Фреймворк для тестирования, обеспечивающий высокое покрытие кода и надежность. Его система мокирования
  идеально подходит для изоляции логики от внешних зависимостей, таких как база данных.

## Настройка окружения

Создайте файл `.env` в корне проекта и заполните его следующими переменными:

```env
#SERVICE SETTINGS
#----------------------------------------
NODE_ENV=local

# RABBITMQ
#----------------------------------------
RABBITMQ_HOST=rabbit1
RABBITMQ_USER=rabbit
RABBITMQ_PASS=rabbit
RABBITMQ_PORT=5672
RABBITMQ_ERLANG_COOKIE=SUPERSECRETCOOKIE

# POSTGRESQL
#----------------------------------------
POSTGRES_USER=postgres
POSTGRES_PASSWORD=supersecretpassword
POSTGRES_DB=sql-alanyzer
POSTGRES_HOST=postgres
POSTGRES_PORT=5432
```

В корневой директории проекта находится [docker-compose.yml](docker-compose.yml) файл, в котором описаны сервисы для
запуска приложения и базы данных. Для запуска приложения необходимо выполнить команду:

```bash
docker-compose up -d
```

После этого, будет поднят экземпляр базы данных PostgreSQL, развернут кластер из трех узлов RabbitMQ и запущено
приложение Sql Analyzer, которое состоит из трех сервисов:

- **producer**: Сервис, который читает файл с SQL запросами и отправляет их в очередь RabbitMQ пакетами для дальнейшей
  обработки.
- **mapper** (в двух экземплярах для демонстарации работоспособности): Сервис, который читает сообщения из очереди
  RabbitMQ, парсит SQL запросы (определяет, к какой таблице был запрос и какие таблицы были join'ы) и сохраняет
  результаты в базе данных PostgreSQL.
- **reducer**: Сервис, который читает результаты из базы данных PostgreSQL и формирует отчет по запросам. Имеет
  небольшой встроенный модуль для выдачи рекомендаций по оптимизации базы данных.

### Миграция базы данных

После запуска приложения необходимо выполнить миграцию базы данных.

Для этого зайдите в контейнер `mapper` и выполните команду:

```bash
docker exec -it mapper sequelize db:migrate
```

## Запуск Producer

Если предыдущий шаг выполнен успешно, сервис `producer` уже запущен.

### Исходные данные

Разумеется, для реальных приложений необходимо получения файл(ов) с SQL запросами через API самого приложения или с
хранилища,
но для демонстрации работы приложения достаточно использование файла [queries.](producer/src/source-data/queries.sql)

При необходимости, вы можете заменить его на свой файл с SQL запросами, но убедитесь, что он соответствует формату,
поддерживаемому приложением (запросы разделены переносами строк).
После этого необходимо пересобрать образ `producer` и запустить с помощью команды:

```bash
docker-compose up -d --build producer
```

### Инициация работы Producer

После того, как файл с SQL запросами готов, необходимо инициировать работу сервиса `producer`, сделав *POST* запрос
*/process* к API приложения.

Для этого можно использовать `curl` внутри контейнера `producer`:

```bash
docker exec -it producer curl -X POST http://localhost:3000/process
```

Выполнение этой команды инициирует процесс обработки SQL запросов, которые будут отправлены в очередь RabbitMQ.

## Запуск Mapper и Reducer

После того, как `producer` отправит SQL запросы в очередь RabbitMQ, сервисы `mapper` автоматически начнут обрабатывать
их. Вы можете проверить логи сервисов `mapper` и `mapper2`, чтобы увидеть процесс обработки сообщений из очереди и
сохранения результатов в базе данных PostgreSQL.

### Получение отчета

Вы можете получить отчет по обработанным SQL запросам, сделав *POST* запрос */process* к API приложения `reducer`.

Для этого можно использовать `curl` внутри контейнера `reducer`:

```bash
docker exec -it reducer curl -X POST http://localhost:4000/recount 
```

После выполнения этой команды, сервис `reducer` сформирует отчет по обработанным SQL запросам и выведёт его в консоль.

## Описание ожидаемого формата входных данных и выходного результата

### Входные данные

Входные данные представляют собой файл с SQL запросами, где каждый запрос находится на отдельной строке.
Пример строки входных данных:

```sql
SELECT "Guild"."id"           AS "Guild.id",
       "Guild"."name"         AS "Guild.name",
       "Guild"."leader_id"    AS "Guild.leader_id",
       "Guild"."server_id"    AS "Guild.server_id",
       "Server"."id"          AS "Server.id",
       "Server"."name"        AS "Server.name",
       "Server"."status"      AS "Server.status",
       "Server"."max_players" AS "Server.max_players",
       "Server"."region_id"   AS "Server.region_id"
FROM "guilds" AS "Guild"
         INNER JOIN "servers" AS "Server" ON "Guild"."server_id" = "Server"."id"
WHERE "Guild"."leader_id" = '918bf2cf-1499-4c17-9f34-fa3093f52001'
LIMIT 50 OFFSET 846;
```

### Выходные данные

Выходные данные представляют собой отчет (в логах), который содержит информацию о количестве запросов к каждой таблице и
количестве join'ов других таблиц в каждом запросе. Пример отчета:

```
Recommendations for table "servers":  
 {
   regions: {
     loadLevel: 'critical',
     percentage: 93.5251798561151,
     recommendations: [
       'Агрессивная денормализация. Рассмотрите возможность переноса всех часто используемых полей из присоединяемой таблицы. Если отношение между таблицами "один к одному", подумайте об их полном слиянии.',
       'Пересмотр архитектуры. Такая сильная связь может указывать на то, что сущности были разделены неправильно. Возможно, с точки зрения логики приложения, это должна быть одна сущность и, соответственно, одна таблица. Этот вариант требует глубокого анализа, но может решить корень проблемы, а не ее симптомы.'
     ]
   }
 }
```

Здесь:

- `servers` - имя таблицы, по которой сформирован отчет.
- `regions` - имя таблицы, которая была присоединена к основной таблице хотя бы один раз. Каждая такая таблица будет
  указана в отчете.
- `loadLevel` - уровень нагрузки на таблицу, который может быть `low`, `medium`, `high` или `critical`.
- `percentage` - процент join'ов к этой таблице относительно общего количества запросов.
- `recommendations` - массив рекомендаций по оптимизации работы с этой таблицей, если уровень нагрузки на нее высокий
  или критический.

## Тестирование

Для тестирования приложения можно использовать встроенные тесты, которые проверяют основные функции сервисов  `mapper` и
`reducer`.
Для запуска тестов перейдите в директорию `mapper` или `reducer` и выполните команду:

```bash
npm run test
```

Это запустит сценарий тестирования *Jest*, находящийся в директории `./jest`, и выведет результаты в консоль.

## Структура проекта и зависимости

Поскольку проект является демонстрационным и направлен, в первую очередь, на выполнение учебной задачи, все три
сервиса (`mapper`, `reducer` и `producer`) размещены в одном репозитори, но в разных директориях.

Каждая директория содержит свой собственные `Dockerfile` и `package.json` файл, что позволяет управлять зависимостями и
и сборкой каждого сервиса отдельно.
